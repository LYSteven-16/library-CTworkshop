Module 2

🧩 Deconstructing — Decomposition

~30 min
Objective: Decompose "Design a library traffic monitoring system" into data collection, transmission, processing, and display modules. Understand decomposition as a core CT concept.

Continuing from Module 1

In Module 1 we defined the problem: "Students need real-time crowding info". But this is huge — how does a sensor become info on a phone?
Like putting an elephant in a fridge — impossible in one step. We need decomposition — breaking big problems into manageable parts.

Onion Model: Layered Breakdown

Peel back the problem layer by layer. Click each layer for details.

Layer 1 🔴 Real Pain Point: No Seats

This is the starting point — the user's experience. Students wander between floors looking for seats.

  • Who's affected? Every student using the library
  • Core: Information asymmetry — students don't know if there are seats
  • Real-time info would solve the problem
Layer 2 🟡 Data Needs: Real-time Count / Floors

To solve this, what data do we need? Turn vague unknowns into specific data requirements.

  • How many now? → Need real-time occupancy
  • Which floor has seats? → Per-floor counts
  • When is it quietest? → Need historical trends
Layer 3 🟢 Functional Modules: Collection → Processing → Display

This is decomposition in action! A large system split into three independent modules, each with clear inputs and outputs.

  • Collection: Sensors read entry/exit signals → output raw data
  • Processing: Algorithm counting + debounce → output accurate count
  • Display: Interface presented to user → output visual info
Layer 4 🔵 Technology: Infrared + Algorithm Design

The core technical layer — what technology implements each module?

  • Collection: IR sensor pairs (IR-A / IR-B) detect blocking order
  • Processing: Counting algorithm + debounce logic (Module 5 deep dive)
  • Display: Mobile App UI, traffic lights + heatmap (designed in Module 4)
