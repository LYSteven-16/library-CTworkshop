Module 2

🧩 Deconstructing — Decomposition

~15 min
Objective: Learn to decompose "Design a library traffic monitoring system" into data collection, transmission, processing, and display modules. Understand decomposition as a core CT concept.

Continuing from Module 1

In Module 1 we defined the problem: "Students need real-time library crowding info". But this problem is huge — how does a sensor become info on a phone?
Like putting an elephant in a fridge, impossible in one step. We need decomposition — breaking big problems into manageable parts.

Onion Model: Layered Problem Breakdown

Peel back the problem layer by layer. The onion diagram below shows the layered structure — click each layer for details.

第 1 層 🔴 Real Pain Point: No Seats Available

This is the starting point — the user's experience. Students wander between floors, wasting time.

  • Who's affected? Every student using the library
  • Core: Information asymmetry — students don't know if there are seats
  • 如果能看到即時資訊，問題就解決了
第 2 層 🟡 Data Needs: Real-time Count / Floors

To solve this, what data do we need? Turn vague unknowns into specific data requirements.

  • How many now? → Need real-time occupancy
  • Which floor has seats? → Per-floor counts
  • When is it quietest? → Need historical trends
第 3 層 🟢 Functional Modules: Collection → Processing → Display

這就是分解的結果！一個大系統被拆成了三個獨立的功能模組，每個都有明確的輸入和輸出。

  • 採集：感測器讀取進出信號 → Output raw data
  • 處理：算法計數+去抖 → Output accurate count
  • 展示：界面呈現給使用者 → Output visual info
第 4 層 🔵 Tech: Infrared + Algorithm Design

The core technical layer — what tech implements each module?

  • Collection: IR sensor pairs (IR-A / IR-B) detect blocking order
  • Processing: counting algorithm + debounce logic (Module 5 deep dive)
  • Display: Mobile App UI, traffic lights + heatmap (designed in Module 4)
